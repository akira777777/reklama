# План непрерывного улучшения reklama

Телеграм-юзербот рассылки (Telethon). Цель: устранить слабые места проекта и
выстроить инкрементальный роадмап, выполняемый фаза за фазой. Каждая фаза
самодостаточна и несёт собственную валидацию, чтобы можно было двигаться
инкрементально и контролируемо.

## Зафиксированные решения

- **Порядок фаз — по severity**, без уклона в одну область: P0 безопасность →
  P0 баги → P1 архитектура → P2 тесты → P3 документация/эксплуатация.
- **Веб-панель (`web.py`, FastAPI :8000) — компонент 1-го класса**: баги чинятся,
  состояние изолируется от CLI, добавляются тесты и раздел в README.
- **Утечка кредов**: ротация (создать новое приложение на my.telegram.org) +
  чистка `.env.example` заглушками. **Историю git не переписывать.**
- **Ядро**: вынести `CampaignEngine` в `reklama/`, убрать глобальный мутабельный
  `control_state`. Единый источник правды для CLI, веба и скриптов.
- **`scripts/forward_from_saved.py`**: превратить в тонкую обёртку над engine
  (либо удалить с заметкой в README). Убрать дублированный send-цикл.

## Фаза 1 — P0 Баги корректности

- **1.1 Сломанные паузы вступления в `search.py`** (строки ~510-529).
  Блок меж-вступительных пауз (`--delay-min/--delay-max`, `--join-batch-*`)
  имеет отступ вне цикла `for` → выполняется один раз *после* цикла вместо
  между вступлениями. Анти-бан логика `--join` фактически не работает.
  Внести блок внутрь цикла `for idx, t in enumerate(to_join)`, чтобы паузы
  срабатывали после каждого вступления (кроме последнего). Заменить голый
  `await asyncio.sleep(...)` на `search_sleep(...)` для корректной остановки и
  обновления `search_state`.
- **1.2 `web.py /api/status`** отдаёт `run.control_state` напрямую и определяет
  статус по строке `"Завершено"`. Заменить на структурированный снимок состояния
  (явные поля + флаг `finished`), без хрупких строковых сравнений.
- **1.3 `web.py` обновление `.env` через `importlib.reload(config)`**
  ненадёжно (значения захвачены в других модулях/локальных переменных во время
  активной кампании). Убрать `reload`; после Фазы 2 чтение конфига пойдёт через
  engine. На время Фазы 1 — явно перечитывать нужные значения в точке
  использования и запретить менять конфиг во время активной кампании.
- **1.4 Race медиа** в `web.py` (`/api/message/media`, DELETE) и
  `import_message.py` (очистка `media/`): ввести `asyncio.Lock` на операции с
  `media/`; запретить менять медиа, пока кампания/поиск активны.

**Валидация:** новый тест `tests/test_search.py` — несколько вступлений
(мок-клиент) → корректное число и длительность `search_sleep`; ручной прогон
веба `--dry-run`; ruff/mypy чисто.

## Фаза 2 — P1 Архитектура (CampaignEngine)

Цель: единый источник правды, устранение дублирования, тестируемость.

- **2.1 `reklama/spintax.py`.** Перенести из `run.py`
  (`SpintaxNode`/`SpintaxText`/`SpintaxChoice`/`resolve_spintax`).
  Обновить импорт в `tests/test_spintax.py` и `scripts/forward_from_saved.py`.
- **2.2 `reklama/engine.py` — `CampaignEngine`.** Инкапсулирует оркестрацию
  рассылки (цикл по группам, паузы, окно активности, адаптивный множитель
  FloodWait, resume по `progress.json`). Состояние — в `dataclass`, **не** в
  глобальном dict. API: `run()/pause()/resume()/skip_delay()/stop()` и
  геттер снимка состояния. **Persist `delay_multiplier`** в `progress.json`,
  чтобы resume сохранял адаптацию к FloodWait (сейчас сбрасывается в 1.0).
  Конфиг передаётся объектом, а не читается из глобалей на ходу.
- **2.3 Разнести `run.py` (731 строка).** Вынести TUI в `reklama/tui.py`
  (`LiveLogHandler`, `make_layout`, `keyboard_listener`, `smart_sleep`).
  `run.py` становится тонким CLI: argparse + создание engine + опциональный TUI.
- **2.4 `web.py` поверх engine.** `WebState.engine: CampaignEngine | None`;
  эндпоинты управления (`/campaign/*`) вызывают методы engine; `/api/status`
  берёт снимок у engine. Прямой доступ к `run.control_state` убрать.
- **2.5 `scripts/forward_from_saved.py`** → тонкая обёртка: `import_message.py`
  (`prepare_message_for_saving`) + `CampaignEngine`. Удалить копию send-цикла.
  Альтернатива — удалить скрипт с заметкой в README (сценарий = import + run).
- **2.6 Убрать мёртвый код.** Проверить использование
  `utils.managed_telegram_client` (дубликат `auth.client_session`); удалить,
  если не используется.

**Валидация:** `mypy .` чисто; `ruff check .` чисто; существующие тесты зелёные;
ручной `--dry-run` через CLI и через веб; прогон `--limit 1` на реальной сессии.

## Риски и открытые вопросы

- **Глобальное состояние** затрагивает веб, CLI и `forward_from_saved`
  одновременно — переход на `CampaignEngine` требует аккуратности; обязательны
  дымовые тесты (`--dry-run`, `--limit 1`) после Фазы 2.
- **`importlib.reload` убирается** (1.3), но изменение конфига через веб в
  рантайме должно продолжать работать — после Фазы 2 это идёт через engine.
- **Ротация кредов (0.1)** — ручное действие пользователя на my.telegram.org;
  невозможно автоматизировать.
- **Удаление `forward_from_saved.py`** (2.5) — если выбрать «удаление», убедиться,
  что ни CI, ни README на него не ссылаются.
- **`media/` race** — блокировка решает веб/импорт, но `run.py` (CLI) и веб могут
  работать с одной папкой одновременно; уточнить, нужен ли cross-process lock
  (для Docker-volume это актуально).

## Порядок выполнения

Фазы идут последовательно (каждая опирается на предыдущую), но внутри фазы
задачи можно делать параллельно. Минимально безопасный путь:
**0 → 1 → 2 → 3 → 4**. После каждой фазы — полный прогон `ruff check .`,
`mypy .`, `pytest` и ручной `--dry-run`.
